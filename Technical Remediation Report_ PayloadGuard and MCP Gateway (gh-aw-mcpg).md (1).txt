Technical Remediation Report: PayloadGuard and MCP Gateway (gh-aw-mcpg) This report outlines the mandatory architectural and security remediations required for the PayloadGuard and MCP Gateway systems. These fixes address critical failures in payload handling, SDK-level pagination, memory management, and observability identified during the recent audit and Test Run ID: 24584250976. -------------------------------------------------------------------------------- #### 1. Remediation of the Large Payload Permission Paradox ##### 1.1 Technical Root Cause Analysis (RCA) Technical Root Cause: Analysis of Test Run ID: 24584250976 identifies a privilege separation conflict between the MCP server's write-context and the agent's read-context. When a payload exceeds the defined size threshold, the MCP server successfully writes the file to the payloadPath. However, because the server process operates within a different security context than the agent, the resulting file is created with restrictive permissions. This prevents the agent from ingesting the large payload, leading to an atomic failure of the tool call despite the file's presence on disk. ##### 1.2 File-System Access Correction To resolve this without resorting to insecure chmod 777 overrides, we must implement a privileged security context using Access Control Lists (ACLs). This ensures that any file created within the payload directory inherits read permissions for the agent while maintaining a hardened posture for the rest of the host environment. Execute the following sequence on the gateway host: bash # Define the internal payload directory PAYLOAD_DIR="/var/lib/mcp-gateway/payloads" # Ensure the directory exists and set ownership to the gateway service user sudo mkdir -p $PAYLOAD_DIR sudo chown -R mcp-gateway:mcp-gateway $PAYLOAD_DIR # Harden directory permissions to allow group read/execute only sudo chmod 750 $PAYLOAD_DIR # Verify the agent's group name (defaulting to 'agent-context') # getent group agent-context || echo "Group not found" # Apply Default ACLs for idempotency; ensure new files inherit read permissions sudo setfacl -d -m g:agent-context:r $PAYLOAD_DIR sudo setfacl -m g:agent-context:r $PAYLOAD_DIR # Verify ACL persistence getfacl $PAYLOAD_DIR  ##### 1.3 Validation Procedure Verification must be performed via the test-harness to confirm the agent's ability to ingest payloads. Resolving SDK Pagination Gaps ##### 2.1 Issue Identification Risk of Silent Tool Truncation: As documented in the Go Fan Report (§23233789121), the current implementation in internal/mcp/connection.go lacks a cursor-based evaluation loop. The system currently only ingests the first page of responses. For backends exceeding single-page responses, this results in a non-deterministic loss of agent capabilities as tools, resources, and prompts are silently truncated at the gateway boundary. ##### 2.2 Implementation of Cursor-Based Loops We must refactor discovery logic to handle NextCursor tokens. The implementation must be applied across the three primary discovery operations: ListTools, ListResources, and ListPrompts. Each operation must loop until result.NextCursor is empty to ensure complete capability discovery. ##### 2.3 Refactored Code Block The following Go implementation demonstrates the pattern required to eliminate truncation risks. Note the subtle differences in result types handled by the single cursor-logic: go func (c *Connection) discoverAll(ctx context.Context) error { // Example: ListTools pagination loop var allTools []sdk.Tool cursor := "" for { res, err := c.client.ListTools(ctx, &sdk.ListToolsParams{Cursor: cursor}) if err != nil { return err } allTools = append(allTools, res.Tools...) if res.NextCursor == "" { break } cursor = res.NextCursor } // Example: ListResources pagination loop var allResources []sdk.Resource cursor = "" for { res, err := c.client.ListResources(ctx, &sdk.ListResourcesParams{Cursor: cursor}) if err != nil { return err } allResources = append(allResources, res.Resources...) if res.NextCursor == "" { break } cursor = res.NextCursor } // Example: ListPrompts pagination loop var allPrompts []sdk.Prompt cursor = "" for { res, err := c.client.ListPrompts(ctx, &sdk.ListPromptsParams{Cursor: cursor}) if err != nil { return err } allPrompts = append(allPrompts, res.Prompts...) if res.NextCursor == "" { break } cursor = res.NextCursor } return nil }  -------------------------------------------------------------------------------- #### 3. Memory Leak Mitigation: filteredServerCache ##### 3.1 Leak Mechanism Description A significant memory leak persists in internal/server/routed.go. The filteredServerCache uses a map key derived from backendID + "/" + sessionID. Because the sessionID is currently tied to transient Authorization headers, the map grows unboundedly with every unique token. While the underlying SDK manages its own SessionTimeout, this external map in routed.go lacks any sweep logic or eviction strategy, leading to non-trivial memory consumption through orphaned tool closures. ##### 3.2 Remediation Steps Architectural health requires two specific interventions: * Max-Size LRU (Least Recently Used) Eviction: Implement a size-bounded cache (default: 1000 entries) to force-evict oldest *sdk.Server instances. * TTL-based Cleanup: Align the cache lifecycle with the 30-minute SessionTimeout. A periodic 5-minute sweep must verify session liveness and purge expired entries. ##### 3.3 Configuration Update The StreamableHTTPOptions must be updated to enforce the following remediated settings: | Setting | Current Setting | Remediated Setting (Architectural Standard) | | ------ | ------ | ------ | | SessionTimeout | 30 minutes | 30 minutes (Strict TTL Enforcement) | | MaxCacheSize | Unbounded | 1000 Entries (LRU Strategy) | | CleanupInterval | None | 5 minutes (Sweep Interval) | -------------------------------------------------------------------------------- #### 4.
